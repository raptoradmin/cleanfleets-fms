Application Project Summary
11/7/2010

I wanted to take some time and make sure we are all on the same page and there is a clear understanding of the current state of the development of the FCMS application.

Below is a table outlining the list of more than 50 requested correction/modifications/additions to the FCMS since 9/29. Needless to say this is a tremendous amount of requests to attempt to process in one month. Several of the requests practically required a complete revision to both the database structure AND the user interface.

The two most significant requests that required considerable time and programming change were:

Creation of the Vehicle Reports with Images (AKA Blue Report)
Creation of an annual Mileage log with Start/End Dates and an annual Hours log with Start/End Dates

Details of these are listed in the table below.

Sean, I understand you are anxious to get going on a Quick Estimate Report so Sean can provide an estimated cost and timeline schedule for current and potential clients. However, most of the items requested within the table below have to be completed BEFORE I can even begin to create such a process. Many of the required fields were not part of the origin specs for the application and were only determined to be required during our meeting of October 7. The revisions requested during that meeting and on the subsequent punch list were significant programming changes. In fact, the application today is almost a completely new application compared to the original requested product.

If you want to see a good comparison of what was the original application and where it stands today, Log into the application as yourself and navigate to any Vehicle Details page that has complete information and images. Then log out and log back in as u=hpotter and p=123ewq@ and Navigate to the first Vehicle Details page. You will see that the application is completely different. The original application's Vehicle Details page had only two tabs, now the application has four. There was no Mileage or Hours log, now they are both present. Keep in mind that every change made to the CF side of the application still needs to be made to the Client side of the application. I still have a lot of programming to go.

I could go on and on about the many changes over the last month, but even creating this Project Summary constitutes considerable time. Suffice to say that I am moving forward as quickly as possible. This is doubly difficult when the programming has to be done on the live product that is being used in production while programming is happening. The reason this is difficult is that I can never leave the application in a state where it is not usable. This is VERY difficult because I have to not only do all the programming of each step of a process, I also have to take the time and test every step one at a time so as not to leave the application in a broken state when I have to stop and work with other clients.

Best practice for programming is to have one version of the application that is specifically used for development and one version that is the production application. No changes should ever made on the production application. Changes should be made to the development application and once the changes have been tested and approved, the development application is PUBLISHED and replaces the production application. (AKA revisioning) Unfortunately we are not working with this method currently so I have to take the extra time to always leave the application in a usable state.

There are risks to this method of development, especially as more and more data is added and clients and CF users become more dependent on the production application. If I make one mistake in creating a process such as DELETE, I could potentially delete ALL the data in the application. This has already happened once as Scott can attest too. (Don’t worry too much about this because I do backup the application and the database every night so the worst that could happen is losing one day’s worth of data entry.) However, you need to be aware of the pitfalls with our method of current programming.

It is very hard to troubleshoot not knowing which form threw the error.
